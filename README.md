## 트러블슈팅
- 마크다운 -> html 할때 타입지정 관련 : api에 한번 보내고 다시 가져오려니 중간중간에 객체의 depth라던지 타입이 바뀌어서 고생했다. 결국 만들었던걸 뜯어내고 next.js/blogstarter에 있는 코드로 뼈대를 갈아서 맞추어주었다. 뼈대의 중요성을 느낀 하루였다.

- 배포 관련 : 리팩토링 이후 깃페이지에서 로딩이 되지 않았다... 깃페이지는 스태틱페이지만 로딩해주는 걸 까먹었다. vercel로 배포를 하자 클릭 몇번에 허무하게 잘되는 것을 확인할 수 있었다. 

- 신기했던 점은 next.js 배포 관련 팁 이라고 써진 블로그에 여러가지 부적같은 코드들이 많았는데 불과 몇달 지난 시점의 나는 아무것도 설정할 필요가 없었다는 것이다. 로컬이랑 웹이미지의 주소를 따로 빼거나, 배포주소에 env 설정으로 로컬/서버를 둘다 등록해준다던가 등등... 빠르게 변화하는 프레임워크라더니 진짜 빠른것 같다. 덕분에 코딩 블로그들을 빨리 포기하고 공식문서와 공식깃헙에 매진할 수 있었다.

- css 관련 : 평소에 tailwind를 즐겨쓰는데 마침 예시코드에서 사용한 css가 tailwind였다. 이걸 가져와버린다면 분명 css를 이걸로 다 때워버릴 것 같아서 바닥부터 설계를 시작했다. 오랜만에 styled-component를 쓰니까 tailwind가 무척 편하고 좋았다는 걸 실감할 수 있었다. 프리셋도 있고, 미디어쿼리도 잘되어있고... 생각나는대로 인라인 css도 편안하게 쓸수 있고...... 하지만 현업에서는 안쓰는 곳이 더 많으니까 styled에도 얼른 다시 익숙해져야겠다. 

- 헤더에 이전글/다음글 추가 : next.js의 단점은 폴더구조가 강제된다는 것이다. api에 쿼리를 달아서 동적으로 불러오기 위해서는 slug이라는 <-- 파일명을 나타내는 값이 필요한데 이 값이 없으면 마크다운 파일을 찾을 수가 없다. 그런데 각 포스트를 불러오는 api에서 이미 `api/[slug]` 주소를 사용해버려서 `slug을 기준으로 앞뒤 포스트를 불러오는 api`를 만들때 경로가 중복되는 문제가 발생했다.

- 쿼리가 값이 되는 구조라 바꿀수도 없고 api를 합치자니 요청시 받아오는 값 때문에 타입을 하나 더 만들기도 좀 그래서... 결국 `[slug]`디렉토리를 하나 만들고 그 뒤에 아무 값 `[featured]`를 사용해서 파일을 하나 더 만들어주었다. 뭔가 해결은 했는데 이거 아닌거 같은데...
